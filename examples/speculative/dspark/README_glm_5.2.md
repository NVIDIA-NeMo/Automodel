# Training a GLM-5.2 DSpark drafter

DSpark draft training for a frozen **GLM-5.2** (`GlmMoeDsaForCausalLM`) target, using the
same semi-autoregressive method and three-term objective as the Qwen3 / Gemma4 / DeepSeek
V4 DSpark configs. The GLM target is loaded frozen via the GLM finetune recipe's
distributed path (expert-parallel + FSDP2): the 256 routed experts shard across ranks and
the FP8 base weights dequantize to bf16 on load. The draft is a small **dense GLM-MLA**
stack (DeepSeek-V3-style Q-LoRA, a compressed KV latent, a separate value up-projection,
interleaved complex RoPE), and hidden states are captured online (no offline cache).

GLM-5.2's own sparse machinery -- the DSA top-k **indexer**, IndexShare layer sharing, and
the **MoE** -- belongs to the target model. The draft drops all of it: every draft layer is
a plain dense MLA + dense SwiGLU block, and it runs the DFlash SDPA (dense-mask) path, not
`flex_attention`.

## Hardware: the full target needs multiple nodes, not one 8x80GB box

GLM-5.2 is a ~355B-parameter MoE (78 layers, 256 routed experts, `hidden_size=6144`). The
DSpark target wrapper hooks **one non-pipelined** `self.model(...)` forward, so pipeline
parallelism must be **off** (`pp_size=1`). Shard the parameter memory with expert
parallelism instead: per-rank expert memory scales as `total_expert_bytes / ep_size`.

- `ep_size` **must divide** `n_routed_experts` (256): 8, 16, 32, 64 are valid.
- `ep_size` should **equal** `world_size` (`nnodes * 8`).
- A larger `ep_size` shards the experts more finely (less memory per rank) but does not
  change the effective data batch: expert parallelism is data-parallel-compatible (each
  rank forwards its own data shard and the MoE all-to-all routes tokens to the
  expert-owning ranks).

The full 78-layer target does **not** fit on a single 8x80GB node; use `ep_size>=32`
(4+ nodes), or `ep_size=64` (8 nodes, the default in `glm_5.2_dspark.yaml`). This is a
resident-weight peak in the expert dequant, **not** fixable by lowering `seq_length` or
enabling `activation_checkpointing`. To sanity-check the whole path on **one** node first,
use the smoke runner (it derives a temporary config with only 6 target layers).

## Prerequisites

- **HybridEP** (or DeepEP) installed. The GLM MoE token dispatcher requires it; without it
  `setup()` raises `ImportError: HybridEP is not installed`.
- For full training: multiple nodes of 8 x >= 80 GiB GPUs (see above). For the smoke: a
  single 8 x >= 80 GiB node (the target is reduced to 4 layers).
- A local GLM-5.2 checkpoint (or the hub id `zai-org/GLM-5.2`). The published config's
  `head_dim: 192` clobbers `qk_rope_head_dim` on load (the HF `attribute_map` maps
  `head_dim -> qk_rope_head_dim`), which would fail checkpoint shape validation; the
  recipe detects this and restores `qk_rope_head_dim` from the raw config automatically,
  so the raw snapshot works as-is.

## 1. Prepare data

DSpark is teacher-forced, so regenerate the responses with the **target** first to avoid a
train/inference distribution mismatch (as in the DSpark paper). Take your prompts, generate
responses with GLM-5.2, and write OpenAI-format `messages` rows to a jsonl:

```json
{"messages": [{"role": "user", "content": "..."}, {"role": "assistant", "content": "<response regenerated by GLM-5.2>"}]}
```

Point `recipe_args.train_data_path` at that jsonl.

**Chat template.** Unlike DeepSeek-V4-Flash, GLM-5.2 ships a chat template in its
tokenizer, so `recipe_args.chat_template` is **not** required -- the `messages` rows render
with the tokenizer's own template. Set `recipe_args.chat_template` only to override it (it
must match the template you used to regenerate the responses, and should mark the assistant
span with `{% generation %}` so only those tokens carry loss).

## 2. Smoke test (run this first, single node)

A fast pipeline-sanity run of the real EP path, with the target shrunk to 6 layers
(`target_num_hidden_layers=6`, i.e. 3 dense layers + 3 routed-MoE / IndexShare layers,
captured feature layers `[3, 4, 5]`) so it fits on **one** 8x80GB node (~24 GiB/rank)
and reaches a step in minutes. It confirms the distributed target
loads, hidden capture works across the shards, the draft trains, and the three-term loss is
finite and trends down:

```
TARGET=/path/to/GLM-5.2 bash tests/functional_tests/speculative/run_glm_5.2_smoke.sh
```

It runs the CPU unit tests, builds 64 tiny smoke rows, derives a temporary reduced config
from `glm_5.2_dspark.yaml`, runs an 8-GPU training smoke, and asserts a finite,
non-increasing loss. A draft trained by the smoke is a pipeline check, not a usable
drafter.

## 3. Full training (multi-node)

Set `target_model_name_or_path` and `train_data_path` in `glm_5.2_dspark.yaml` first (it
defaults to `ep_size=64` for 8 nodes). Run the **same** command on every node, varying only
`--node-rank`; node 0 is the rendezvous master and `<NODE0_IP>` is its address reachable
from the other nodes:

```
torchrun --nnodes=8 --node-rank=0 --nproc_per_node=8 --master-addr=<NODE0_IP> --master-port=29500 -m nemo_automodel.recipes.llm.train_dspark -c examples/speculative/dspark/glm_5.2_dspark.yaml
```

Run the same line on nodes 1..7 with `--node-rank` 1..7. For a 4-node run set `ep_size: 32`
in the yaml and `--nnodes=4` with `--node-rank` 0..3. If inter-node NCCL stalls on a fabric
without routed RDMA, force the TCP socket path on every node:
`NCCL_IB_DISABLE=1 NCCL_SOCKET_IFNAME=<iface>`. Per-step metrics (`loss`, `ce_loss`,
`l1_loss`, `confidence_loss`, `lr`, `mem`) are written to
`<output_dir>/dspark_train_metrics.jsonl`.

### Attention backend

`target_attn_backend` defaults to `sdpa` (the model's dense Torch attention path): reliable
everywhere and correct -- the DSA indexer emits an additive float bias that SDPA's
explicit-mask path accepts. Set it to `tilelang` for the fused GLM DSA indexer / sparse-MLA
kernels **if** your TileLang build JIT-compiles them.

## Validation status

- **Verified**: CPU unit tests for the draft / config builder / registry and the RoPE
  `freqs` fp32 pin across a bf16 cast (`tests/unit_tests/speculative/test_dspark_glm_5_2.py`).
- **Smoke-validated on 8x H100 80GB**: the reduced 6-layer EP8 path (SDPA attention,
  `torch_mm` experts, activation checkpointing) loads the FP8 checkpoint, captures
  hidden states across the shards, and trains end to end with a finite three-term
  loss at ~24 GiB/rank. This is a pipeline check, not a convergence claim.
- **Needs a first multi-node run**: the full 78-layer distributed target load and
  cross-shard hidden capture at `ep_size>=32`. Points to eyeball during it:
  1. FP8 base weights dequantize to bf16 on load (target forward stays finite).
  2. Captured hidden states are full per-rank (not a sharded DTensor that needs an all-gather).
  3. The frozen GLM target forward runs with a 2D mask and `use_cache=False`.
  4. `ep_size` divides `n_routed_experts` (256); 64 gives 4 experts/rank.
  5. Per-rank weight memory fits (watch `nvidia-smi`); drop `ep_size` / add nodes if not.
