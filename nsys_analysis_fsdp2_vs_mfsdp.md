## Why AutoModel FSDP2 is Faster

### 1. torch.compile → `nvjet_sm90` Hopper kernels
The profile contains 31,245 GEMM invocations of `nvjet_sm90`-family kernels — these are AOT-compiled, Hopper-native tensor-core kernels generated by torch.compile. They out-perform standard cuBLAS/TE GEMM by better utilizing H100 SM90 features. MFSDP runs without compile and uses standard TE GEMM.

### 2. Near-perfect ReduceScatter overlap (97%)
RS follows the backward pass, which produces dense compute (FlashAttn bwd + GEMM). Only 0.856 s of RS is left exposed. PyTorch FSDP2's prefetch hooks achieve near-ideal RS hiding.

### 3. AG exposure is a structural limit, not a bug
91% of the 2.574 s AG exposure (2.351 s) comes from >500 µs kernels at layer boundaries — specifically the first layer's AllGather at the start of each grad-accumulation step, where there is no prior-layer computation to overlap with. This is a fundamental constraint, not an implementation deficiency.

### 4. AllToAll penalty for EP=1 (dense model)
MFSDP's `ep1_alltoall` variant routes tokens through an MoE-style AllToAll dispatch/combine path even though there are no actual experts (EP=1). Every token goes to every "expert" on the same device, adding routing overhead with zero benefit. AllToAll is designed for EP>1; forcing it onto a dense model degrades performance.

## Lessons from the MFSDP AllToAll Profile (even as the slower run)

### Lesson 1: AllToAll as a universal comm primitive is a tradeoff
MFSDP explores a single AllToAll code path that handles both dense and MoE models uniformly. For EP=1 the dispatch/combine overhead is pure waste. For EP>1 (real MoE) this amortizes expert routing cost across the same kernel.  
**Takeaway**: Keep AllGather/ReduceScatter for dense models. AllToAll only pays off when actual token redistribution across experts occurs (EP>1).

### Lesson 2: Recompute granularity — from xuwenc's benchmark log, not nsys
From xuwenc's `benchmark_results.md` Key Findings (not derived from the nsys profile): `recompute-num-layers=10` is reported as **7.6% slower** than `=1` (381.3 vs 412.6 TFLOPs/GPU). However, those two rows also differ in `manual-ubr` presence, so the delta is not a clean recompute-only comparison. The nsys profile would be needed to confirm whether the gap is caused by recompute granularity or the ubr flag.  
**Takeaway (tentative)**: Per-layer recompute appears preferable, but this needs a controlled experiment to isolate the recompute variable.

### Lesson 3: manual-ubr instability — from xuwenc's benchmark log, not nsys
From xuwenc's `benchmark_results.md` Key Findings (not derived from the nsys profile): `manual-ubr` works for the TP=1 CP=2 config but fails for TP=2 CP=1, with the root cause stated only as "NCCL error 801 on CW cluster nodes." The exact reason (topology, NCCL version, config bug) is not determined from the available data.  
**Takeaway**: manual-ubr is unstable on this cluster; don't enable it without further investigation.

### Lesson 4: RS is inherently easier to overlap than AG
RS exposure (0.856 s, 3%) < AG exposure (2.574 s, 10%) even under ideal conditions. The backward pass produces dense, sustained compute that hides RS naturally. AG precedes forward compute where the queue is thinner (especially at the first layer). This asymmetry is fundamental — future AG overlap improvements have diminishing returns compared to RS.

### Lesson 5: FP32 gradient reduction is not the performance bottleneck
Both runs use FP32 grad reduce (2× the RS data volume vs BF16), yet AutoModel is still faster. With RS at 97% overlap, even doubling the RS data volume doesn't materially hurt throughput. This validates choosing FP32 for numerical stability without sacrificing performance.

### Lesson 6: HSDP two-level AllGather can create synchronization boundaries
MFSDP's HSDP=1 decomposes AllGather into intra-node + inter-node phases. The boundary between the two phases introduces an additional synchronization point not present in FSDP2's flat AllGather. The nsys profile (once exportable) would reveal this as two distinct AG kernel bursts per layer vs one in AutoModel.

**Bottom line**: AutoModel FSDP2 wins primarily through torch.compile (Hopper-native GEMM kernels) and highly effective FSDP2 prefetch (97% RS overlap, 90% AG overlap). The MFSDP AllToAll variant is the weakest configuration — it validates that AllToAll overhead hurts dense-model FSDP when EP=1. Even standard MFSDP runs ~2% behind, driven by the absence of compile-time kernel optimization.

## Appendix: How to Export MFSDP SQLite (future reference)

The MFSDP profile requires nsys ≥ 2026.1.2. Options:
1. Ask xuwenc to re-export using `gitlab-master.nvidia.com/xuwenc/docker_pytorch:pytorch26.03_te2.14_hybridep_x86` on a node with that image cached.
2. Update one of your containers to include nsys ≥ 2026.1.2, then run:
   ```bash
   nsys export --type sqlite --output mfsdp.sqlite --force-overwrite true \
     MFSDP-llama3-70b/llama3_70b_mfsdp_tp1_pp1_ep1_cp2_hsdp1_alltoall_mbs1_gbs128_seqlen8192_cw_n8_full-recompute.nsys-rep
   ```
3. Then run `analyze_allgather_overlap.py mfsdp.sqlite` to get the AllToAll exposure metrics.
